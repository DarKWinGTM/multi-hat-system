# Anti-Patterns

This file collects the most common failure modes in a multi-hat analysis package.

---

## Mode Anti-Patterns

- decision mode used without explicit comparison intent
- advisory mode polluted by unnecessary option generation
- clarify-first used to stall instead of to resolve decisive uncertainty

---

## Hat Anti-Patterns

- one hat doing all the real work while the others are decorative
- security comments reduced to generic “use best practices”
- architecture comments reduced to trend language
- developer comments reduced to stack preference without delivery reality

---

## Synthesis Anti-Patterns

- three stacked reports and no resolution
- “it depends” without explaining what it depends on
- recommending the safest-sounding option without delivery realism
- recommending the fastest path while hiding long-term cost

---

## Communication Anti-Patterns

- fake certainty
- vague blocker language
- recommendation with no trade-off statement
- comparison with no clear winner

---

## Structural Anti-Patterns

- making hats into separate runtime agents
- letting support files compete with runtime authority
- packing all handbook material into one oversized file when it should be modularized
- reintroducing archive material as an active dependency
